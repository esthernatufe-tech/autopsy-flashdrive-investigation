## Overview

This project documents a complete digital forensics investigation of a USB flash drive using Autopsy 4.21.0. The investigation followed standard forensic phases — acquisition, examination, analysis, and reporting — to recover deleted files, analyse file metadata, identify suspicious artefacts, and reconstruct a timeline of user activity.

The case demonstrates practical skills in:
- Digital evidence acquisition and preservation
- Deleted file recovery from unallocated disk space
- File metadata extraction and analysis
- Timeline reconstruction of user activity
- Keyword search and suspicious file identification
- Formal forensic report writing

## Investigation Methodology

The investigation followed the standard four-phase digital forensics model:

**1. Preparation & Case Setup**
A new case was created in Autopsy with a unique case ID, examiner details, and a designated base directory. Evidence handling procedures were documented to ensure forensic integrity.

**2. Evidence Acquisition**
The USB flash drive was added as a Local Disk data source. Autopsy generated a logical copy of the drive's file system (NTFS/FAT) to prevent any modification of the original evidence. All ingest modules were enabled before analysis began.

**3. Examination & Analysis**
Autopsy automatically ran all ingest modules and surfaced results across multiple categories including deleted files, keyword hits, metadata, and timeline events. Each finding was manually verified by reviewing timestamps, file paths, and metadata.

**4. Documentation & Reporting**
Findings were tagged inside Autopsy and a formal HTML forensic report was generated. A written analysis report was produced following academic and professional forensic standards.

## Key Findings

### Deleted File Recovery
- **362 deleted files** were successfully recovered from unallocated disk space
- File types recovered include Excel spreadsheets, batch scripts, executables, DLLs, PowerPoint files, Word documents, and images

| File Type | Count |
|-----------|-------|
| Excel (.xlsx / .xls) | 96 |
| Batch scripts (.bat) | 83 |
| Executables (.exe) | 71 |
| DLL files (.dll) | 26 |
| Data files (.dat) | 24 |
| Temp files (.tmp) | 14 |
| PowerPoint (.pptx) | 4 |
| Word documents (.docx) | 2 |
| Other | 42 |

### Notable Recovered File — HADIZA CR-NEW.xlsx
- **File path:** `/img_D:/__/HADIZA CR-NEW.xlsx`
- **File type:** Microsoft Excel Spreadsheet
- **Created:** 2023-09-28 | **Last Modified:** 2021-04-26 | **Last Accessed:** 2023-10-24
- **Metadata allocation:** Unallocated (confirmed deleted)
- The metadata indicated deliberate deletion rather than accidental loss, based on timestamps and file path patterns. The file was tagged for further examination as a potentially sensitive artefact.

### Suspicious Files — 6 Flagged
Six files were flagged as suspicious based on irregular activity timestamps, naming inconsistencies, and metadata anomalies. One file in particular — **AL ITQAAN.docx** — was marked as "Likely Notable" by Autopsy's keyword search module because it contained an embedded email address, triggering an Email Addresses keyword hit.

### Timeline Analysis
Autopsy's timeline feature revealed a clear pattern of user activity:
- **Blue activity bars** = normal file creation, access, and modification events
- **Red activity bars** = suspicious deletion and modification events

A concentration of red (deletion) activity was observed **immediately before the flash drive was disconnected**, strongly suggesting deliberate attempts to erase data prior to removal of the device.

### Geolocation Analysis
No EXIF geolocation metadata was found in any recovered images or documents, indicating that no files were captured or saved using GPS-enabled devices.

## Limitations Identified

- Autopsy did not automatically detect the flash drive in some instances, requiring manual path selection
- High RAM and CPU consumption during ingest of large data volumes
- Limited support for encrypted or severely corrupted media
- No native support for mobile device or cloud storage forensics
- Open-source status may raise admissibility concerns in formal legal proceedings without supporting documentation
